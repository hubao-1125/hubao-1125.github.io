面对海量会员数据，如何处理深分页（Deep Pagination）导致的性能瓶颈？

约 1149 字大约 4 分钟

面试

2026-05-12

海量会员的深分页，标准解法是 ES+search_after、游标分页、异步导出、预计算 这四类。 深分页的本质是：用 offset 跳页时，数据库必须扫描并丢弃前面所有数据，越往后扫得越多、越慢。

分库分表后，情况更糟：

  • 你要查 LIMIT 100000,10，每个分片都得先查前100010条（不知道别的分片有啥）；
  • 所有分片数据拉到应用层全局排序、裁切，极易OOM、超时；
  • 数据量越大、分片越多，深分页越慢，分库分表解决的是“存储容量”，不是“深分页算法复杂度”。

一句话：分库分表 ≠ 免深分页；深分页该踩的坑一个都不会少，还会多跨片开销。

一、海量会员深分页的标准解决方案（4类，按优先级）

1. ES+search_after 游标分页（最优，会员列表首选）

核心：不用 offset，用“排序字段+唯一主键”当游标，每次从游标位置往后取，性能恒定 O(size)，和页码无关。

  • 排序规则：create_time desc + member_id desc（时间+唯一ID，防重复）
  • 第一页：正常查，取最后一条的 [create_time, member_id] 作为游标
  • 下一页：把游标塞进 search_after，ES 自动从该位置往后取10条
// 第一页请求
{ "size": 10, "sort": [{"create_time":"desc"},{"member_id":"desc"}] }

// 第二页请求（带上一页最后一条的排序值）
{
  "size": 10,
  "sort": [{"create_time":"desc"},{"member_id":"desc"}],
  "search_after": ["2026-05-12 14:30:20", 10099]
}
  • 优点：性能稳定、无偏移、支持全量顺序翻页
  • 缺点：不支持跳页（只能上一页/下一页），符合会员列表场景

2. MySQL 游标分页（无ES时用）

原理同 search_after，用 WHERE 排序字段 < 最后一条值 替代 offset。

-- 第一页
SELECT * FROM member 
ORDER BY create_time DESC, member_id DESC 
LIMIT 10;

-- 第二页（用第一页最后一条的时间+ID）
SELECT * FROM member 
WHERE create_time < '2026-05-12 14:30:20' 
   OR (create_time = '2026-05-12 14:30:20' AND member_id < 10099)
ORDER BY create_time DESC, member_id DESC 
LIMIT 10;
  • 优点：改动小、性能接近ES
  • 缺点：排序字段必须有序+唯一组合，跨分片需额外归并

3. 大数据量导出：异步任务+文件生成（必做）

会员导出（如百万级）绝对不能同步分页查：

  • 同步查：数据库压力大、超时、OOM
  • 正确做法：异步任务（如XXL-JOB）→ 分批查（每次1000条）→ 写入CSV/Excel → 上传OSS → 通知用户下载

4. 统计/报表：预计算层（避免实时聚合）

  • 实时查：COUNT(*)、多条件聚合、排序，千万级数据极慢
  • 正确做法：定时任务（每小时/每日）预计算会员数、新增数、地区分布 → 存入汇总表/ClickHouse → 报表查预计算结果

二、关键问题：排序值到底是什么？代码怎么存？

1. 排序值（游标）= 排序字段值数组

  • 格式：[排序字段1值, 排序字段2值, ...]
  • 会员场景：[create_time, member_id]（时间+唯一ID）
  • 例子：["2026-05-12 14:30:20", 10099]

2. 代码怎么存？—— 前端临时存，后端不持久化

  • 后端：第一页查询后，从结果最后一条提取排序值，随列表返回给前端
  • 前端：存在内存/状态里（如Vue/React的变量），下一页请求时原样带回
  • 后端收到：直接塞进 search_after 或拼接 SQL 条件，不用查库、不用Redis

3. 能用 name 当排序值吗？

  • 可以语法上：sort: [{"name":"keyword"}] → search_after: ["张三"]
  • 但强烈不推荐：name 重复会导致漏数据、重复数据、定位不准
  • 必须：普通排序字段 + 唯一主键（如 name + member_id）

4. 排序值需要建索引时指定吗？

  • 不需要在建索引时特殊定义
  • 只要字段是：keyword/date/数值型（可排序），且查询时写在 sort 里，就能当排序值

三、分库分表+深分页的正确姿势（不是不用，是要配合游标）

  1. 分片键选 member_id（查询带 member_id 路由到单分片，无跨片）
  2. 全局会员列表：同步到ES，用search_after（避开跨片深分页）
  3. 必须跨片时：禁止 offset，用游标分页 + 分片结果归并

四、一句话总结

  • ❌ 错误：分库分表 = 不用深分页（反而更糟）
  • ✅ 正确：海量会员深分页 → ES+search_after 游标分页（列表）+ 异步导出（大数据量）+ 预计算（报表）
  • 核心：抛弃 offset，用“排序字段+唯一主键”当游标，前端传、后端用，性能恒定
