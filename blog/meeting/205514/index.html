在设计分布式缓存架构时，如何保证数据库与缓存的一致性？请对比不同方案（如先删后写、延时双删）的优缺点及适用场景。

约 1060 字大约 4 分钟

面试

2026-05-12

一、先讲主流三种缓存一致性方案

1. 先更新数据库、再删缓存（普通删更）

流程： 更新DB → 立刻删除缓存

优点

  • 逻辑最简单、代码少、易维护
  • 读多写少场景够用

缺点

  • 并发下有脏数据： A更新DB还没删缓存，B查缓存读到旧数据，写入缓存，造成长期脏数据
  • 无法解决读写并发冲突

适用场景

  • 普通非核心业务、一致性要求不高、读多写少

2. 延时双删（先删 + 写完再删 + 延迟再删一次）

流程：

  1. 先删缓存
  2. 更新数据库
  3. 延迟几百毫秒再删一次缓存

优点

  • 能解决读写并发带来的旧数据缓存残留问题
  • 比单纯先删后写一致性好很多

缺点

  1. 延时时间不好把控，依赖预估业务耗时
  2. 会频繁删缓存，增加缓存、DB查询压力
  3. 第一次删完，请求回填新缓存，延时二次删除又把新缓存删掉，造成缓存频繁击穿，大量请求打数据库
  4. 依赖定时/线程池做延迟，架构不优雅

适用场景

  • 中小型项目、没有引入中间件、要折中一致性

3. 给缓存加过期时间（万能兜底）

不管用哪种更新策略，都必须加缓存TTL 就算出现脏数据，到期自动失效，最终能自愈。

二、工业级生产主流方案：Canal 监听 Binlog 异步更新缓存（大厂标配）

流程

  1. 业务只只管更新数据库，不操作缓存
  2. Canal 监听 MySQL binlog
  3. 感知到数据变更，异步删除/更新缓存

优点

  1. 业务代码零侵入，不用手动写删缓存逻辑
  2. 异步解耦，性能高
  3. 彻底避开读写并发、双删延时不准的问题
  4. 天然最终一致性
  5. 削峰，不会瞬时大量删缓存压垮Redis

缺点

  • 需要部署Canal组件，架构稍重

适用场景

  • 支付、会员、订单等核心分布式业务
  • 高并发、强一致性、不允许脏数据

三、还有强一致方案：分布式锁 + 更新后删缓存

流程： 更新DB和删缓存全程加分布式锁，串行化操作

优点：强一致性，无脏数据 缺点：并发性能下降，锁竞争 适用：一致性极高、并发不高的核心数据

四、三种方案横向对比

  1. 先更DB后删缓存：简单易开发，一致性弱，适合普通非核心业务；并发易产生脏缓存。
  2. 延时双删：解决读写并发脏数据，不用中间件；但延时难把控、易缓存击穿、增加DB缓存压力。
  3. Canal监听Binlog：业务无侵入、异步解耦、最终一致性稳定，是分布式架构主流方案；缺点是需要额外部署中间件。 同时所有方案都必须配置缓存过期时间做兜底自愈。

五、精简

保证缓存与数据库一致性，我会分三种方案选型： 第一种是先更新数据库再删除缓存，实现简单适合普通业务，但并发场景容易产生脏缓存。 第二种是延时双删，写完数据后延迟再删一次，能解决读写并发脏数据问题；但延时不好把控，容易频繁删除缓存造成缓存击穿，增加数据库压力。 生产分布式项目主流用Canal监听Binlog异步删缓存，业务只操作数据库，由Canal异步感知变更更新缓存，代码无侵入、解耦高性能，保障最终一致性；同时所有方案都配置缓存TTL过期做兜底，防止长期脏数据。需要强一致性的场景再搭配分布式锁串行化操作。
