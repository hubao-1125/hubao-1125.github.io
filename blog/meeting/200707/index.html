在高并发场景下，如何根据业务特征选择垃圾收集器并调优 JVM 参数以平衡吞吐量和延迟？

约 708 字大约 2 分钟

面试

2026-05-12

一、高并发下 GC 选择只有 3 种，按业务场景对号入座：

1. 吞吐量优先（后台批处理、计算任务、数据同步）

Parallel GC（JDK8 默认） 适合：

  • 后台任务
  • 数据计算
  • 批处理
  • 不追求响应时间，只求快干完

特点：

  • 多核并行收集
  • STW 停顿较长
  • 高吞吐量

关键参数：

-Xms4g -Xmx4g
-XX:+UseParallelGC
-XX:MaxGCPauseMillis=200  # 最大停顿毫秒数

2. 平衡型（Web 服务、支付、会员、订单，90%业务）

G1 GC适合：

  • 微服务
  • 支付中台
  • 会员系统
  • 订单服务
  • 高并发 Web 服务

特点：

  • 兼顾吞吐量 + 低延迟
  • 可控停顿
  • 堆内存大也能用

关键参数：

-Xms4g -Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200  # 目标停顿200ms
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用45%开始GC

3. 低延迟优先（高频交易、实时通信、强响应要求）

ZGC / Shenandoah（几乎无停顿） 适合：

  • 支付核心链路
  • 高频交易
  • 延迟敏感（10ms 内）

特点：

  • 亚毫秒级停顿
  • 几乎无 STW
  • 占一点点 CPU

关键参数：

-Xms4g -Xmx4g
-XX:+UseZGC
-XX:ZCollectionInterval=30  # 30s扫描一次

二、高并发下 JVM 调优的通用原则（面试官最爱听）

  1. 堆内存固定-Xms = -Xmx 避免扩容收缩

  2. 新生代不能太小 避免频繁 Young GC

  3. GC 停顿目标明确 普通业务：200ms 实时业务：50ms内

  4. GC 耗时不能超过总运行时间的 1% 超过就说明内存或GC选择不合理

三、满分回答

在高并发场景下，我们根据业务特征选择垃圾收集器：

  1. 后台批处理、吞吐量优先的业务，采用 Parallel GC，追求高任务处理效率，通过 -XX:MaxGCPauseMillis 控制最大停顿时间。
  2. 微服务、支付、会员、订单等 Web 高并发业务，采用 G1 GC，平衡吞吐量与延迟，设置目标停顿时间 200ms，兼顾响应速度与稳定性。
  3. 低延迟、强实时核心业务，采用 ZGC，实现亚毫秒级停顿，保证核心链路不被GC影响。

JVM 调优方面，我们固定堆内存 -Xms=-Xmx 避免伸缩，合理调整 GC 触发阈值，通过监控 GC 频率、暂停时间，保证 GC 耗时不超过总运行时间的 1%，最终达到吞吐量与延迟的平衡。

四、最精简

我们根据业务选择GC：吞吐量优先用Parallel，通用高并发用G1，低延迟用ZGC。 JVM 调优固定堆内存，设置合理停顿目标，通过GC监控保证暂停时间在可控范围，平衡吞吐量与延迟。
