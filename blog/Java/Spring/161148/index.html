Transactional 注解失效的常见场景

约 1469 字大约 5 分钟

面试题Spring

2026-06-24

一、底层核心原理（先搞懂为什么会失效）

Spring 事务基于 AOP 动态代理 实现：

  1. 容器创建 Bean 时，若类 / 方法标注@Transactional，会生成代理对象；

  2. 外部调用代理对象带事务的方法时，代理会拦截方法，在方法执行前开启事务、执行后提交 / 回滚；

  3. 只有外部通过代理对象调用，AOP 拦截逻辑才会生效；

  4. 如果是内部 this 调用、无代理、代理无法拦截，事务增强逻辑完全不会执行，注解直接失效。

代理分两种：

  • JDK 动态代理：目标类实现接口，代理实现接口；
  • CGLIB 代理：目标类无接口，生成子类代理原类。

二、@Transactional 失效全部常见场景 + 原因解析

场景 1：同类内部方法调用（this 自调用）

示例代码：

@Service
public class UserService {
    public void outer() {
        // this是原始对象，不是代理对象，不会触发AOP拦截
        this.inner();
    }
    @Transactional
    public void inner() {
        // 数据库操作异常，事务不会回滚
    }
}

失效原因： this指向原生对象，而非 Spring 生成的代理对象，AOP 事务拦截器无法切入，事务逻辑不执行。 解决方案：

  1. 自身注入UserService，用代理对象调用：@Autowired UserService self; self.inner();

  2. 拆分方法到不同类；

  3. 使用ApplicationContext获取代理 Bean。

场景 2：注解标注在非 public 方法上

@Transactional
private void updateUser() {}

失效原因： Spring AOP 只能拦截public方法；CGLIB 子类重写也只能重写 public 方法，private/protected/final 方法无法被代理拦截，事务逻辑不生效。 解决方案：事务方法统一定义为public。

场景 3：方法被 final/static 修饰

@Transactional
public final void update() {}

失效原因： CGLIB 通过生成子类重写目标方法实现代理，final方法无法被子类重写，无法添加事务增强；static 属于类本身，不属于实例，代理无法拦截。 解决方案：事务方法去掉 final、static 修饰符。

场景 4：没有被 Spring 容器管理（类未加 @Service/@Component）

// 无@Service，不会生成代理
public class UserService {
    @Transactional
    public void save(){}
}

失效原因： 不在 Spring IoC 容器中的类，不会创建代理对象，AOP 事务逻辑不会加载。 解决方案：业务类添加@Service/@Component交给 Spring 管理。

场景 5：多线程 / 异步调用事务方法

@Service
public class UserService {
    public void test() {
        new Thread(()->{
            saveUser();
        }).start();
    }
    @Transactional
    public void saveUser(){}
}

失效原因： Spring 事务绑定在当前主线程的 ThreadLocal，新线程无事务上下文，无法继承父线程事务，新方法会新开独立事务 / 无事务。 解决方案：异步方法单独添加事务，或通过事务传播机制手动处理。

场景 6：异常类型不匹配（默认只捕获 RuntimeException/Error）

@Transactional
public void biz() throws Exception {
    throw new Exception(); // 受检异常，不会回滚
}

失效原因： @Transactional默认仅在抛出RuntimeException、Error时触发回滚；普通受检 Exception 不会触发回滚。 解决方案：指定回滚异常：@Transactional(rollbackFor = Exception.class)

场景 7：异常被 try-catch 捕获，未向上抛出

@Transactional
public void biz() {
    try {
        // 数据库操作报错
    } catch (Exception e) {
        // 异常被吞，没有抛出给代理拦截器
    }
}

失效原因： AOP 拦截器依靠方法抛出异常感知是否需要回滚；异常被内部捕获不抛出，代理无法感知异常，直接提交事务。 解决方案：catch 后重新抛出异常，手动事务回滚TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();

场景 8：事务传播行为配置错误

例：@Transactional(propagation = Propagation.SUPPORTS) 若外层无事务，当前方法不会开启事务，报错不回滚。 失效原因：传播属性控制事务创建逻辑，部分属性在无外层事务时不新建事务。 解决方案：读写业务统一使用REQUIRED（默认）。

场景 9：数据库引擎不支持事务（MyISAM）

失效原因：MyISAM 引擎本身不支持事务、回滚，无论注解如何配置都无效。 解决方案：数据表引擎改为 InnoDB。

场景 10：Spring 事务管理器未配置（缺少 PlatformTransactionManager Bean）

失效原因：缺少事务管理 Bean，Spring 无法初始化事务切面，注解无任何逻辑。 解决方案：引入spring-boot-starter-jdbc/mybatis自动注入事务管理器；纯 Spring 手动配置DataSourceTransactionManager。

场景 11：使用 this 获取代理对象错误（同场景 1 补充）

和内部调用本质一致，原生对象无代理，事务拦截失效。

三、快速记忆总结

  1. 代理层面失效（最常见） 内部 this 调用、非 public/final/static 方法、类未交给 Spring 管理 → 代理无法拦截，事务逻辑不执行；

  2. 异常规则失效 受检异常未指定 rollbackFor、异常被 try-catch 吞掉 → 代理感知不到异常，不回滚；

  3. 环境 / 配置失效 MyISAM 引擎、缺少事务管理器、传播属性配置错误、多线程异步调用。

四、面试精简口述版

底层原理

Spring 事务基于 AOP 动态代理实现，只有外部调用代理对象的 public 方法，才会触发事务拦截、开启 / 回滚事务；如果调用走原生对象，代理逻辑失效，注解无效。

高频失效场景

  1. 同类内部 this 调用，原生对象无代理；

  2. 方法非 public、final/static 修饰，无法被代理重写；

  3. 异常被 try-catch 捕获不抛出，或抛出受检异常未配置 rollbackFor；

  4. 数据库 MyISAM 引擎不支持事务；

  5. 类未注入 Spring 容器，不生成代理；

  6. 多线程异步调用，事务 ThreadLocal 无法跨线程传递；

  7. 传播行为配置错误、缺少事务管理器 Bean。
